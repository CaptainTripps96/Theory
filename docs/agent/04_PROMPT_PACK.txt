# TheorySequencer Context-Window-Friendly Prompt Pack

Use one prompt at a time in VS Code with your AI coding agent.

After each prompt:
1. Let the AI edit/create files.
2. Build.
3. Run tests.
4. Manually verify the requested behavior.
5. Commit the working state.
6. Move to the next prompt.

Do not let the AI skip ahead. Do not ask it to build the whole DAW in one pass.

Working assumptions:
- App name: TheorySequencer
- Project package extension: .tseq
- C++ namespace: tsq
- Language: C++20
- Framework: JUCE
- Audio/sequencing backend: Tracktion Engine
- Build system: CMake
- Testing: Catch2
- First plugin format: VST3 instruments only
- IDE/editor: VS Code

============================================================
UNIVERSAL RULES FOR EVERY CODING PROMPT
============================================================

You are working on TheorySequencer, a commercial-quality cross-platform C++20/JUCE/Tracktion Engine MIDI sequencer and future DAW.

Do not optimize for speed at the cost of architecture. Do not create throwaway code unless explicitly asked. Do not bypass the architecture.

Core rules:
- Use C++20.
- Use CMake.
- Use JUCE for app/UI framework.
- Use Tracktion Engine behind our own playback adapter layer.
- Use Catch2 for core tests.
- Keep core music/sequencer logic independent from UI and audio backend.
- UI must not directly mutate project state; use commands.
- Audio thread must not allocate, lock, parse JSON, access files, call UI code, or do expensive theory analysis.
- VST3 instruments are required early.
- Do not implement audio recording, AU, VST2, CLAP, AAX, staff notation, Roman numeral analysis, or full mixing unless specifically asked.
- Do not allow overlapping MIDI clips on the same track.
- Do not automatically change MIDI pitches when key/scale/mode changes.
- Used accidental lanes remain visible per clip.
- Chord overlays are visual only and must not restrict playback or note entry.
- Update docs/IMPLEMENTATION_LOG.md after each prompt with a concise summary of changes, commands run, tests passed/failed, and known issues.
- If an API differs from your assumptions, inspect the local dependency headers/examples and adapt. Do not invent APIs.
